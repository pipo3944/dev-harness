# Explain ドキュメント 内容チェックリスト

このドキュメントは「理解せず進む」を防ぐためのもの。HTML + canvas の図は
あくまで理解の補助であり、装飾ではない。以下の中身が全て揃って初めて合格。

バグ修正にも新機能にも使う。**「問題」は新機能では「その変更が解消する痛み /
今できていないこと」と読み替える**。以下のチェックリストは内部構造の解説枠ではなく、
「主役（作るもの/直すもの）を最初に、内部理屈は最後に」という順序で読める doc を作るためのもの。

## 構成の鉄則（順序）— これを外すと読みにくくなる

1. **最初に「作るもの / ゴール」を大きく置く**。新機能なら完成イメージ、バグなら
   直後のあるべき挙動。**内部構造・アーキ図・実装メカニズムから始めない。**
2. **問題は必ず具体シナリオで示す** — 具体的な人物・具体的な入力値・具体的な結果の
   物語（「こう操作して → こう壊れる」「今これをやろうとすると → こう詰まる」）。
   抽象説明だけで済ませない。
3. **なぜ問題か → 原因 → アプローチ → なぜ解決するか** を、そのシナリオを軸に展開する。
4. **内部構造 / 実装詳細 / 周辺タスクは末尾に畳む**（読みたい人向け。`<details>` 等）。
   内部モデル変換・ライブラリ既定挙動・周辺の事故は脇役。前面に出さない。

## 必須セクション（上の順序に沿って埋める）
- [ ] 作るもの / ゴール — 何を実現するか（新機能: 完成イメージ / バグ: あるべき挙動）
- [ ] 具体シナリオ — 誰が・どんな入力で・どうなるか（今の痛み or 壊れ方の物語）
- [ ] なぜ問題か（本質） — そのシナリオがなぜ起きるのか、根本原因
- [ ] 何が足りない / 間違っているか — 具体
- [ ] どう変えれば実現 / 解決するか — 変更方針（before/after のイメージ、確定した approach）
- [ ] 影響範囲 — やると何に波及するか
- [ ] （末尾に畳む）該当コード・内部構造 — file:line + 抜粋、実装メカニズム

## 視覚化の指針（canvas/図）
- [ ] シナリオ駆動 — 図は「なぜ問題が起きるか / なぜこの変更で解決するか」を
      具体例で説明していること。**アーキ図単体で始めない**（見た目だけの装飾にしない）。
- [ ] 状態遷移・シーケンス・データフローのうち、理解に効くものを選んで図化
  - 例: 「build 時に now() を1回評価 → 時刻経過しても再評価されない」を状態遷移で

## 合格ライン
- 専門外の人が読んでも「何を作る/直すのか、どう実現するか」が掴める
- 主役（作るもの/直すもの）が前面にあり、内部理屈が主役を埋もれさせていない
- かつ「なぜそうなるのか」の本質まで分かる
