# Explain ドキュメント 内容チェックリスト

このドキュメントは「理解せず進む」を防ぐためのもの。HTML + canvas の図は
あくまで理解の補助であり、装飾ではない。以下の中身が全て揃って初めて合格。

## 必須セクション
- [ ] 問題の所在 — どの画面/機能/フローの話か
- [ ] 現状の挙動 — 今どう動くか（ユーザーから見える形で）
- [ ] 該当コード — どのファイルのどの記述が原因か（file:line + 抜粋）
- [ ] なぜ問題か（本質） — その記述だとなぜこうなるのか、根本原因
- [ ] 何が足りない/間違っているか — 具体
- [ ] どう変えれば直るか — 変更方針（before/after のイメージ）
- [ ] 影響範囲 — 直すと何に波及するか

## 視覚化の指針（canvas/図）
- [ ] 状態遷移・シーケンス・データフローのうち、理解に効くものを選んで図化
- [ ] 図は「なぜ問題が起きるか」を説明していること（見た目だけの装飾にしない）
  - 例: 「build 時に now() を1回評価 → 時刻経過しても再評価されない」を状態遷移で

## 合格ライン
- 専門外の人が読んでも「何が問題で、どう直すか」が掴める
- かつ「なぜそうなるのか」の本質まで分かる
