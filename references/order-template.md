## ワーカー向け注意

- **report の前に、最終保存版に対して** format / analyze / 当該テストを通すこと。
  「green」は自己申告になりがちで、保存とコマンド実行のズレ（整形し忘れ等）が残ると
  オーケストレーターの独立検証や CI で落ちる。
- **検証の誠実さ（最重要）**:
  - **実行していないコマンドの結果を書かない**。実行できなかったら「できなかった」を
    理由つきで書く。それは減点ではない — **捏造だけが致命傷**。
  - report の検証結果は**オーケストレーターが独立検証で必ず裏取りする**。照合される
    前提で書くこと。
  - 検証コマンドは**パイプ（`| tail` / `| head` / `| grep`）に通さず素で実行**し、
    各コマンドの **exit code を確認**する（パイプすると exit code が末尾コマンドの
    ものになり失敗を隠す）。
  - 「動作確認」には**コマンド・exit code・末尾の生出力をそのまま貼る**
    （要約・記憶で書かない）。
- **コードコメントは自己完結させる**。ハーネス内部の ID（フェーズ番号・レビュー
  finding ID = `M9`/`N2` 等）や approach 内の呼称（`案B` 等）をコメントに書かない。
  *なぜそうしたか*（理由）を書く。

## 完了時アクション

- 本ディレクトリに **report.md** を作成。ワーカーは dev-harness スキルを起動して
  いないので `references/...` の相対パスは解決できない。**下の見出し骨子をそのまま
  使う**こと（オーケストレーターは order を書くとき、テンプレ名を転記せず、この骨子を
  インライン展開して渡す）:

  ```markdown
  # Report: phase-NN {タイトル}
  ## サマリ            … 何をやったか3行以内
  ## 変更点            … path:line — 何をどう変えたか
  ## 受け入れ条件の検証  … [x]/[ ] 各条件をどう確認したか。未達は理由
  ## 遭遇した問題・判断  … 問題 → どう判断したか / なぜ
  ## 残課題・申し送り    … 次フェーズ or follow-up。なければ「なし」
  ## 動作確認          … 実測のみ。コマンド → exit code → 末尾の生出力を貼る
  ```

- コミットはオーケストレーターがフェーズゲート承認後に行う（ワーカーは勝手に
  コミットしない）
- オーケストレーターへ完了報告
