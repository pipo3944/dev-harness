# Order: アーキ批判レビュー（approach 設計レビュー / step 3）

> オーケストレーター → レビューサブエージェント。実装*前*の設計レビュー。
> approach.md（＋explain／関連実コード／リポジトリ）をフレッシュな文脈で読み、
> 「このフレームワークの作法として最善か」を**敵対的に**指摘する。
> step7 の実装後・差分レビューとは別物（こちらは設計の筋を問う）。

## 対象
- approach: .dev/{dir}/approach.md
- 関連コード: path:line ...
- issue: #{issue}

## レビュー観点
対象の言語/フレームワークを diff・コードから判定し、その慣習と repo の既存流儀に照らすこと。

- [ ] 状態管理の置き場 — その値/ロジックは適切なレイヤーにあるか
- [ ] ライフサイクル / 破棄 — リソース・購読・タイマーの生成と破棄が対になっているか
- [ ] 再描画 / 再計算コスト — 過剰な再構築や無駄な再取得を生まないか
- [ ] テスト容易性 — 純関数に切り出せる所が UI 等に混ざっていないか
- [ ] 既存パターンとの一貫性 — repo に既にある流儀/抽象と揃っているか
- [ ] 影響範囲の網羅性 — 同じ原因の箇所を取りこぼしていないか（横展開漏れ）
- [ ] よりよい代替設計 — その FW の定石として、もっと筋の良い案はないか

## 出力（approach-review/report.md）
- 観点ごとに: 指摘 / 重大度 / 根拠（file:line・FW 慣習） / 改善提案
- 「現状の approach で良い」場合はその旨（指摘ゼロも有効な結論）
- 総評: approach をどう修正すべきか（あれば具体案）

## 注意
- 敵対的に（「もっともらしい」で素通りさせない）。ただし根拠なき難癖はしない。
- 実装はしない。設計の評価と代替提案まで。
